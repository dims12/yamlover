# yamlover-desktop

An Electron shell around the yamlover server. It spawns the **same** server that
ships to npm (`tools/server`) in production mode, bound to `127.0.0.1`, and opens a
window at it. No Vite at runtime ÔÇö it serves the prebuilt `dist/`. The web UI's live
file-update flow (FS watcher Ôćĺ `/api/events` SSE) works unchanged inside the window.

## How it works

- `main.js` builds the server (`vite build` + esbuild Ôćĺ `tools/server/dist`), then
  `utilityProcess.fork`s `tools/server/bin/yamlover.js <root> --prod`. The server is
  run in **Electron's bundled Node**, which is why this app requires an Electron
  whose Node is **Ôëą 22.5** (the engine's store uses `node:sqlite`). Electron 35+
  ships Node 22.x ÔÇö see the version floor in `package.json`.
- The launcher prints its URL on stdout; `main.js` reads it and `loadURL`s the
  window, so the port-in-use fallback is honored automatically.
- The chosen folder is remembered in `userData/config.json`; **File Ôćĺ Open FolderÔÇŽ**
  (Ôîś/Ctrl-O) switches trees.

## Run in dev

From this directory:

```
npm install        # electron + electron-builder
npm start          # builds tools/server, then launches the window
```

(`npm start` runs `build:server` first, so `tools/server/dist` is always fresh.)

## Build installers

```
npm run dist       # current OS: .dmg (mac) / .nsis (win) / .AppImage (linux)
npm run dist:dir   # unpacked app dir only (faster, for smoke-testing)
```

`electron-builder` copies `tools/server`'s `bin/`, `dist/` and `package.json` into
the app under `resources/server` (the `package.json` is needed so Node treats the
ESM `bin`/`dist` as modules). Cross-OS builds follow electron-builder's usual rules
(build each target on its own OS, or via CI).

## Icon

`build/icon.png` (1024├Ś1024) is generated by `build/make-icon.mjs` (pure Node, no
image deps) ÔÇö electron-builder derives the per-OS `.icns`/`.ico` from it. To change
the artwork, edit and re-run `node build/make-icon.mjs`, then commit the PNG.

## CI

`.github/workflows/desktop.yml` builds all three installers on GitHub
(`macos`/`windows`/`ubuntu-latest`). It runs on **`workflow_dispatch`** (artifacts on
the run) and on a **`v*` tag** push (also attaches the installers to that tag's
GitHub Release). The desktop version is synced from `tools/server/package.json`, so
keep bumping only the server version.

Trigger a manual build (needs the `gh` CLI authenticated to the GitHub mirror):

```
gh workflow run desktop.yml            # or use the Actions tab Ôćĺ Run workflow
```

## Code-signing (optional)

Builds are **unsigned** unless signing secrets are present (macOS/Windows then warn
on first launch). To sign, add these GitHub repo secrets ÔÇö the workflow already wires
them in, and their absence is a no-op:

- `CSC_LINK` / `CSC_KEY_PASSWORD` ÔÇö base64-encoded `.p12` certificate + its password.
  Signs **both** macOS and Windows.
- `APPLE_ID` / `APPLE_APP_SPECIFIC_PASSWORD` / `APPLE_TEAM_ID` ÔÇö enables macOS
  **notarization** (required for distribution outside your own machines). The
  hardened-runtime entitlements are in `build/entitlements.mac.plist`.
